# 로딩 시간 단축을 위한 리팩터링 검토

## 1. 현재 로딩 경로 요약

### 1.1 데이터 흐름
- **클라이언트**: `GET /bank|card|cash/api/category-applied-data` (또는 `processed-data`) 호출
- **서버**:
  1. 캐시 없으면 `load_category_file()` → `safe_read_data_json(path)` 로 **전체 파일 읽기 + json.load()**
  2. 필터(은행/날짜/계좌 등) 적용
  3. `df.to_dict('records')` → **전체 행을 list of dict로 변환**
  4. `_json_safe(data)` → **전체 재귀 순회** (NaN/NaT 등 치환)
  5. `jsonify({..., 'data': data})` → **한 번에 JSON 직렬화**
- **클라이언트**: `response.json()` 후 `data.data` 전체로 테이블 HTML 생성 → `innerHTML` 한 번 설정

### 1.2 병목 지점
| 구간 | 병목 | 데이터 규모에 따른 영향 |
|------|------|-------------------------|
| 파일 읽기 | `json.load(f)` 전체 로드 | 파일 크기 ∝ 메모리·파싱 시간 |
| 캐시 미스 | ensure + classify 등 | 첫 요청 시에만, 백그라운드·프리웜으로 상당 부분 완화됨 |
| 직렬화 | `to_dict('records')` + `_json_safe` + `json.dumps` | 행 수 ∝ CPU 시간 |
| 네트워크 | 한 번에 큰 JSON 전송 | 페이로드 크기 ∝ 전송 시간 |
| 클라이언트 | `JSON.parse` + 전체 행으로 HTML 문자열 생성 + DOM 한 번에 삽입 | 행 수 ∝ 파싱·렌더 시간 |

## 2. 적용된 개선 (현재 상태)

| 항목 | 내용 | 상태 |
|------|------|------|
| 백그라운드 생성·프리웜 | 서버 기동 시 bank/card/cash JSON 선행 생성(Bank·Card 병렬) 후 메모리 캐시 적재 → 첫 요청 시 파일 I/O 생략 | (적용됨) |
| 캐시 정책 | 재생성 버튼 시에만 캐시 무효화, mtime 재검사 없음 | (적용됨) |
| xlsx → JSON 전환 | bank/card/cash before·after를 xlsx 대신 JSON으로 저장. Excel 의존 제거, 파싱 속도·호환성 개선 | (적용됨) |
| .bak 파일 미생성 | 백업 파일을 만들지 않고, 필요 시 파일 삭제 후 재생성 또는 빈 DataFrame 반환 | (적용됨) |
| 테이블 페이지네이션 | JSON 테이블 출력 시 30건 단위 페이지네이션으로 첫 화면 응답 크기 감소 | (적용됨) |

## 3. 캐시 동작: 첫 로딩 vs 다음 로딩

전처리 페이지 진입 시 프론트는 다음 순서로 요청한다.

| API | 용도 | 은행 | 신용카드 |
|-----|------|------|----------|
| `/api/source-data` | 전처리전 테이블 | ✅ | ✅ |
| `/api/processed-data` | 전처리후 테이블 | ✅ | ✅ |
| `/api/category-applied-data` | 카테고리 적용후 테이블 | ✅ | ✅ |

### 3.1 은행거래 (MyBank) 캐시

| 캐시 변수 | 내용 | 무효화 시점 |
|-----------|------|-------------|
| `_source_bank_cache` | .source/Bank 엑셀 → 전처리전 데이터(리스트) | 재생성 버튼 |
| `_bank_before_cache` | bank_before.json → DataFrame | 재생성 버튼 |
| `_bank_after_cache` | bank_after.json → DataFrame | 재생성 버튼 |

**첫 로딩** (캐시 전부 비어 있음)
1. `source-data` — `.source/Bank` 엑셀 전부 디스크에서 읽기·파싱 → 캐시 저장 (느림: 엑셀 I/O)
2. `processed-data` — bank_before.json 없으면 엑셀 읽기→전처리→JSON 쓰기→JSON 읽기 → 캐시 (느림: 엑셀+JSON I/O)
3. `category-applied-data` — bank_after.json 읽기 → 캐시 (느림: JSON I/O)

**다음 로딩** (캐시 있음) — 세 API 모두 메모리 캐시에서 필터만 적용해 반환. **디스크 접근 없음**.

### 3.2 신용카드 (MyCard) 캐시

| 캐시 변수 | 내용 | 무효화 시점 |
|-----------|------|-------------|
| `_source_card_cache` | .source/Card 엑셀 → 전처리전 데이터(리스트) | 재생성 버튼 |
| `_card_before_cache` | card_before.json → DataFrame | 재생성 버튼 |
| `_card_after_cache` | card_after.json → DataFrame | 재생성 버튼 |

**첫 로딩** — 은행과 동일 패턴: 엑셀 읽기→통합·전처리→JSON 쓰기/읽기 → 캐시.
**다음 로딩** — 캐시 히트, 디스크 접근 없음.

### 3.3 속도 차이 요약

| 구분 | 첫 로딩 | 다음 로딩 |
|------|---------|-----------|
| 전처리전(source) | .source 엑셀 **전부 디스크에서 읽기·파싱** | **메모리 캐시**에서 필터만 |
| 전처리후(before) | 엑셀 읽기 + 전처리 + JSON 쓰기 + **JSON 읽기** | **메모리 캐시**만 사용 |
| 카테고리 적용후(after) | **JSON 읽기** (bank_after/card_after) | **메모리 캐시**만 사용 |

데이터가 클수록 첫 로딩이 더 느려지고, 다음 로딩은 캐시만 쓰므로 상대적으로 매우 빠르다.

### 3.4 캐시가 비워지는 경우
- **재생성 버튼** 클릭 시: 캐시 변수 전부 `None` 초기화
- **서버 재시작**: 프로세스 종료로 모든 캐시 소멸

이후 첫 요청은 다시 "첫 로딩"과 동일하게 디스크에서 읽고 캐시를 채운다.

## 4. 미적용 리팩터링 제안

### 4.1 API 페이지네이션 확장 (limit/offset)
**내용**: `category-applied-data` / `processed-data`에 `limit`·`offset` 지원. 캐시 DataFrame에서 필터 적용 후 `df.iloc[offset:offset+limit]`만 직렬화, `total`은 전체 행 수 반환.

- **효과**: 첫 응답 크기·파싱·렌더 시간 감소, 체감 로딩 시간 단축.
- **비용**: API 스펙 확장, 프론트 페이지/스크롤 처리 추가.

### 4.2 첫 화면만 빠르게 + 전체는 백그라운드
**내용**: 첫 요청은 `limit=500` 등 고정 작은 값만 반환, 클라이언트에서 "전체 로드" 버튼 또는 자동으로 전체 재요청.

- **효과**: 첫 페이지만 빠르게 그리기.
- **비용**: 페이지네이션과 유사. "전체" 요청 시 큰 응답 한 번 발생.

### 4.3 JSON 파싱·직렬화 가속 (orjson)
**내용**: `json.load()` → `orjson.loads()`, `json.dumps` → `orjson.dumps()`. Flask `jsonify()` 직렬화를 orjson JSONProvider로 교체.

- **효과**: 대용량 JSON 기준 파싱·직렬화 시간 수~수십 % 단축.
- **비용**: 의존성 추가, 기존 `json` 사용처와 호환 유지 필요.

### 4.4 _json_safe 경량화
**내용**: `df.where(pd.notna(df), None)` 후 `to_dict('records')`만 하고, `_json_safe`를 생략하거나 list of dict만 한 번 순회하며 NaN/numpy/datetime만 치환하는 경량 버전으로 대체(재귀 제거).

- **효과**: 대용량 data 리스트 전체 재귀 순회 제거 시 CPU 절감.
- **비용**: NaN/NaT/numpy 타입이 직렬화 단계에서 예외 가능성 있으므로 최소 치환 설계 필요.

### 4.5 응답 압축 (Gzip)
**내용**: Flask `after_request`에서 응답 body를 gzip 압축 전송.

- **효과**: 대용량 JSON 시 네트워크 전송량·전송 시간 감소.
- **비용**: 서버 CPU 소폭 증가, 클라이언트는 브라우저가 자동 해제.

### 4.6 클라이언트 가상 스크롤
**내용**: DOM에 "보이는 구간 + 여유분"만 두고, 스크롤 시 기존 배열에서 슬라이스해 행 갱신.

- **효과**: 1만 건 이상 시 DOM 노드 수 감소, 스크롤·리플로우 부담 감소.
- **비용**: 전체 데이터를 한 번 받아야 하므로 "첫 로딩" 시간은 그대로.

### 4.7 스트리밍 전송
**내용**: `Transfer-Encoding: chunked`로 JSON 배열을 청크 단위로 전송, 클라이언트에서 청크별 파싱·행 추가.

- **효과**: 첫 청크 도착 후 바로 그리기 시작, 체감 지연 감소.
- **비용**: API·프론트 양쪽 변경 폭이 크고, JSON 스트리밍 파싱(NDJSON 등) 설계 필요.

### 4.8 API 타임아웃
**내용**: 프론트 fetch 타임아웃을 60초 등으로 제한해 무한 로딩 방지. 모든 앱에서 동일 적용 권장.

### 4.9 캐시·조건부 요청
**내용**: 테이블 데이터에 `Last-Modified`/`ETag` 또는 짧은 캐시(5초)를 두어 동일 데이터 재요청 시 304 또는 로컬 캐시 사용.

### 4.10 저장 JSON indent 선택
**내용**: 대용량 저장 시 indent 없음 옵션(쓰기·파일 크기·읽기 소폭 개선).

## 5. 우선순위 제안

| 순위 | 항목 | 기대 효과 | 구현 난이도 |
|------|------|-----------|-------------|
| 1 | Flask 응답 orjson (4.3) | API 응답 생성 시간 감소 | 하 |
| 2 | _json_safe 경량화 (4.4) | 대용량 data CPU 감소 | 중 |
| 3 | 응답 Gzip (4.5) | 대용량 시 전송 시간 감소 | 하 |
| 4 | API 페이지네이션 확장 (4.1) | 첫 화면 로딩 대폭 감소 | 중 |
| 5 | 가상 스크롤 (4.6) | 대량 행 스크롤·렌더 부담 감소 | 중 |
| 6 | 스트리밍 (4.7) | 체감 지연 추가 감소 | 상 |

## 6. 정리
- **이미 적용**: xlsx→JSON 전환, .bak 미생성, 30건 페이지네이션, 백그라운드 생성·캐시 프리웜·Bank/Card 병렬로 "첫 요청 시 파일 생성/읽기" 부담은 크게 줄어든 상태.
- **다음 단계로 유력한 것**: **orjson + _json_safe 경량화**로 서버 응답 생성 비용을 줄이고, 필요 시 **Gzip + 페이지네이션 확장**으로 전송·렌더 비용을 추가로 줄이는 구성이 균형이 좋음.
