## 2. 미적용 잔여 제안

### 2.1 API 일관성

**`load_category_table` vs `get_category_table` 반환값 혼동**

- **현재**: `load_category_table(path)` → DataFrame 또는 None (한 개 값), `get_category_table(path)` → (DataFrame, file_existed) (튜플)
- **문제**: 호출부에서 `df, existed = load_category_table(p)` 같은 실수 시 `ValueError` 발생. 반환 형태가 달라 혼동 가능.
- **제안**:
  - 두 함수의 역할을 문서화하고, "조회만 할 때는 `get_category_table`, df만 필요할 때는 `load_category_table`" 사용 규칙 정리.
  - 또는 `load_category_table`에 `return_file_existed: bool = False` 인자를 두고 통일하는 방식도 검토 가능.

### 2.2 예외 처리 정교화

**`except Exception: pass` 줄이기**

- **현재**: 많은 곳에서 모든 예외를 무시. 디버깅 시 원인 파악이 어려움.
- **제안**:
  - 최소한 로깅 추가: `logging.exception(...)` 또는 `print(f"[WARN] ...: {e}", flush=True)`.
  - "예상되는 예외"만 잡고 그 외는 `raise`로 전파하거나 상위에서 일괄 로깅.
- **적용 범위**: bank_app/card_app은 일괄 제안에서 일부 적용 완료. process_bank/card/cash_data 등 나머지 구간이 미적용.

**category_table_io 내부 구체적 예외**

- **현재**: xlsx 마이그레이션 시 `except Exception: pass`로 실패를 삼킴.
- **제안**: `FileNotFoundError`, `PermissionError`, `pd.errors.EmptyDataError` 등 구체적 예외만 잡고, 나머지는 로그 후 재발생 또는 "마이그레이션 스킵"만 명시적으로 처리.

### 2.3 경로·입력 검증

**`safe_write_category_table` 진입 검증**

- **현재**: `df`에 `CATEGORY_TABLE_COLUMNS`가 없을 수 있으나 검사 없음.
- **제안**: 진입 시 필수 컬럼 존재 여부 검사, 없으면 `ValueError`로 실패. df가 None일 때도 `raise ValueError("df is required")` 추가.

**`get_category_table_path` 빈 project_root 처리**

- **현재**: `project_root`가 빈 문자열이면 상대 경로가 나올 수 있음.
- **제안**: 빈 문자열을 `None`과 동일 처리하거나, `os.path.normpath(os.path.abspath(...))` 로 정규화.

### 2.4 가독성 개선

**load_category_table 마이그레이션 조건**

- **현재**: `if not os.path.exists(path) or (os.path.exists(path) and os.path.getsize(path) == 0):` — 논리적으로 맞지만 가독성이 떨어짐.
- **제안**: `file_missing_or_empty = not os.path.exists(path) or os.path.getsize(path) == 0` 변수로 의도를 드러내기.

**migrate_bank_category_file zip 오류 분기**

- **현재**: JSON 전용인데도 `_is_bad_zip_error(e)` 분기가 남아 있음.
- **제안**: 현재는 JSON만 사용하므로 제거하거나, "현재는 JSON만 사용하므로 이 분기는 거의 타지 않음" 주석 추가.

### 2.5 중복 제거

**process_cash/card_data ImportError 폴백 통일**

- **현재**: `category_table_io` 실패 시 각 모듈에서 긴 fallback 정의.
- **제안**: `category_table_fallback`을 사용하도록 통일하면 폴백 로직이 한 곳만 존재.

### 2.6 lib 타입 힌트·docstring 보강

- **현재**: `lib/path_config.py`, `lib/data_json_io.py` 등 공개 함수에 반환 타입·Args/Returns docstring 없음.
- **제안**: `get_data_dir() -> str`, `safe_read_data_json(path, default_empty=True) -> pd.DataFrame | None` 등 점진적으로 추가. 새로 손대는 함수부터 적용.

### 2.7 환경

**.source/Cash 디렉터리 자동 생성**

- **현재**: app.py에서 `.source/Bank`, `.source/Card`만 생성. `.source/Cash`는 미생성.
- **제안**: Cash 사용 시 오류 방지를 위해 앱 기동 시 `.source/Cash`도 `makedirs(..., exist_ok=True)`.

### 2.8 경로·상수 중앙화 (path_config 확장)

- **현재**: `BANK_AFTER_PATH`, `CARD_AFTER_PATH`, `CASH_AFTER_PATH`, `SOURCE_BANK_DIR`, `SOURCE_CARD_DIR` 등을 각 앱과 process_* 모듈에서 따로 정의. `lib.path_config`에는 `DATA_DIR`, `get_category_table_json_path()` 등만 있음.
- **제안**: `lib.path_config`에 `get_bank_after_path()`, `get_card_after_path()`, `get_cash_after_path()`, `get_source_bank_dir()`, `get_source_card_dir()` 추가. 각 앱·process 모듈은 이를 import.
- **효과**: 경로 규칙 변경 시 수정 1곳. 난이도 낮음.

### 2.9 after 캐시 패턴 공통화

- **현재**: bank_app, card_app, cash_app에서 동일한 캐시 패턴 반복 (`_*_after_cache`, `_*_after_cache_mtime`, mtime 비교 후 캐시 반환/무효화).
- **제안**: `lib/after_cache.py`에 팩토리 함수로 "파일 경로에 대한 캐시 get/invalidate 담당 클래스 또는 클로저" 제공. 세 앱은 각자 인스턴스 하나씩만 보유.
- **효과**: 동일 로직 한 곳 관리. 난이도 중 (전역 변수 제거와 호출부 교체 필요).

### 2.10 process 모듈 로딩 방식 통일

- **현재**: 은행 `import process_bank_data as _pbd` (일반), 카드 `importlib.util.spec_from_file_location` (동적).
- **권장**: 카드도 일반 `import process_card_data as _pcd`로 통일 (옵션 A). 코드 단순화·일관성 확보. 동적 로딩의 역사적 이유가 없다면 통일이 유리.

### 2.11 ensure/generate 시 중복 읽기 제거

- **현재**: ensure에서 df를 확보한 뒤 `classify_and_save()`에 넘기지 않는 경로가 있으면, classify_and_save() 내부에서 다시 `integrate_bank_transactions()` 호출 → 동일 통합/읽기가 두 번 발생 가능.
- **제안**: 이미 확보한 df가 있으면 반드시 `classify_and_save(input_df=df)`로 넘기고, 내부에서는 "input_df 없을 때만" 읽기. 카드도 동일.
- **효과**: 대용량 원본 시 통합/읽기 1회로 감소. 난이도 낮음.

### 2.12 cash_after 오류 보관 방식 통일 (선택)

- **현재**: bank/card는 실패 시 `LAST_CLASSIFY_ERROR` 전역 변수 설정. cash에는 전역 오류 변수 없음.
- **제안**: MyCash에 `LAST_MERGE_ERROR` 등을 두고, 실패 시 오류 메시지 설정. reset/API에서 bank/card와 동일하게 사용.
- **효과**: 세 앱의 오류 처리 방식 통일. 난이도 낮음.

### 2.13 문서화 (코드 변경 최소)

- **제안**: 데이터 플로우 문서에 아래 내용 보강:
  - 데이터 플로우: 은행/카드는 process_* 한 모듈에서 source→before→after. 금융정보는 bank_after+card_after 병합만, .source/Cash 미사용.
  - 캐시: 세 앱 모두 after JSON을 메모리 캐시(mtime 기반)로 보관, 재생성/clear 시 무효화.
  - 경로: data/*.json, .source/Bank, .source/Card는 path_config 또는 각 앱 상수로 정의.
