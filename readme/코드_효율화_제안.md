## 1. 경로·상수 중앙화 (path_config)

**현상**  
- `BANK_AFTER_PATH`, `CARD_AFTER_PATH`, `CASH_AFTER_PATH`, `SOURCE_BANK_DIR`, `SOURCE_CARD_DIR` 등을 각 앱(bank_app, card_app, cash_app)과 process_* 모듈에서 따로 정의.  
- `lib.path_config`에는 `DATA_DIR`, `get_category_table_json_path()` 등만 있고, after·source 경로는 없음.

**제안**  
- `lib.path_config`에 다음 함수(또는 상수) 추가:
  - `get_bank_after_path()`, `get_card_after_path()`, `get_cash_after_path()`
  - `get_source_bank_dir()`, `get_source_card_dir()`
- 각 앱·process 모듈은 위를 import해서 사용. `DATA_DIR` 기반이므로 `MYRISK_ROOT` 등 환경 변경 시 한 곳만 수정.

**효과**  
- 경로 규칙 변경 시 수정 지점 1곳.  
- 난이도: 낮음. 기존 경로 fallback(PROJECT_ROOT/data/...) 유지 가능.

## 2. after 캐시 패턴 공통화 (lib)

**현상**  
- bank_app, card_app, cash_app에서 **동일 패턴** 반복:
  - `_*_after_cache = None`, `_*_after_cache_mtime = None`
  - 로드 시: 파일 존재·mtime 확인 → 캐시 hit이면 copy 반환, miss면 JSON 읽어 캐시 저장 후 반환
  - 무효화: 재생성/clear 시 캐시=None, mtime=None

**제안**  
- `lib/after_cache.py` (또는 `lib/cache_after.py`) 추가:
  - `make_after_cache(file_path_key)` 같은 팩토리로, “파일 경로 문자열을 받아서, 해당 경로에 대한 캐시 get/invalidate만 담당하는 클래스 또는 클로저” 제공.
  - get(): (path, safe_read_fn) → df 또는 빈 DataFrame. 내부에서 mtime 비교 후 캐시 반환 또는 읽기.
  - invalidate(): 캐시·mtime 초기화.
- bank/card/cash 앱은 각자 “자기 after 경로”로 위 캐시 인스턴스 하나씩만 두고, 기존 `load_category_file()`/`get_processed_data()` 등은 그 인스턴스의 get/invalidate 호출로 대체.

**효과**  
- 동일 로직 한 곳에서 관리, 버그 수정·mtime 정책 변경 시 한 번만 수정.  
- 난이도: 중. 기존 전역 변수 제거와 호출부 교체 필요.

## 3. process 모듈 로딩 방식 통일

**현상**  
- 은행: `import process_bank_data as _pbd` (일반 import).  
- 카드: `_load_process_card_data_module()`로 `importlib.util.spec_from_file_location` 사용.

**제안**  
- **옵션 A**: 카드도 은행처럼 일반 `import process_card_data as _pcd`로 통일. (이미 같은 프로세스·같은 cwd에서 동작하므로 동적 로딩 필요성 낮음.)  
- **옵션 B**: 은행도 카드처럼 동적 로딩으로 통일. (process 모듈 오류 시 앱 자체 import 실패를 막고 싶을 때 유리.)

**권장**  
- **옵션 A**. 코드 단순화·일관성 확보. 카드만 동적 로딩한 역사적 이유가 없다면 통일하는 쪽이 유지보수에 유리.

## 4. ensure / generate-category 시 중복 읽기 제거

**현상**  
- 은행: `ensure_all_bank_files()` → `_ensure_bank_after_and_category_only()`에서 `integrate_bank_transactions()` 또는 기존 bank_after 파일 읽기 → `classify_and_save(input_df=df)` 호출.  
- `classify_and_save()` 내부에서 input_df가 없으면 다시 `integrate_bank_transactions()` 호출.  
- “이미 ensure에서 df를 확보했는데, classify_and_save()에 넘기지 않는 경로”가 있으면 같은 통합/읽기가 두 번 일어날 수 있음.

**제안**  
- ensure 흐름을 점검해, **이미 확보한 df가 있으면 반드시 `classify_and_save(input_df=df)`로 넘기고**, classify_and_save() 내부에서 “input_df 없을 때만” integrate/파일 읽기 하도록 명확히 유지.  
- 카드도 동일: ensure/재생성 시 이미 integrate로 만든 df가 있으면 `classify_and_save(input_df=df)`에 넘겨서 이중 호출 방지.

**효과**  
- 대용량 원본 시 통합/읽기 1회로 줄어듦.  
- 난이도: 낮음. 호출부만 정리.

## 5. cash_after 반환·오류 보관 방식을 bank/card와 맞추기 (선택)

**현상**  
- bank/card: `classify_and_save()` → (success, error, count), 실패 시 `LAST_CLASSIFY_ERROR` 설정.  
- cash: `merge_bank_card_to_cash_after()` → (success, error, count) 반환은 동일하나, 전역 `LAST_*` 없음.

**제안**  
- MyCash에서 `LAST_MERGE_ERROR`(또는 `LAST_CASH_AFTER_ERROR`) 같은 전역 변수를 두고, `merge_bank_card_to_cash_after()` 실패 시 해당 변수에 오류 메시지 설정.  
- reset/API에서 “금융정보 병합 오류 상세” 표시 시 bank/card와 동일하게 사용 가능.

**효과**  
- 세 앱의 오류 처리 방식 통일.  
- 난이도: 낮음.
